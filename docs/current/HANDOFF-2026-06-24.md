## 未决 / 下一步（按优先级）

### A. 产品定位升级（重要，影响后面一切）
用户明确：报告要**信息密度高、内容详尽**——逐条需求分析、排雷、岗位要求、简历指导、求职指导全包，目标是**替代花钱报班**。推论：
- 工作流输出可能要更详尽（逐条需求拆解，不是几段概述）。
- 风格必须能**装大量文字**（所以下面 B 风格不行）。

### B. 风格定调（当前卡点）
- 已画 B「简报/批注感」mockup → 用户嫌**太简约**，不符合「内容很多」。
- **待画 A「报告/密件感」、C「杂志/观点感」给用户对比。** 约束：能承载详尽逐条内容、信息密度高、但不廉价、不堆 SaaS 卡片（用户讨厌 `identical card grids` = impeccable 的 absolute ban）。
- **标题决策**：页面标题用「岗位 + 公司」（如"丁香园 · 医药AI产品经理"），**不要**用"一块跳板"这种总起句（用户不读总起句）；判断结论放报告**内部**醒目区。

### C. 接前端 ②③（风格定调后）
- ② 输入页：简历上传 + JD 输入 + 用户意图采集（目标方向/绝不接受，现硬编码在脚本里）+ **OCR 识图**（BOSS 等不让复制 JD，用户已申请百度识图 API，待接）。
- ③ 结果页：照定的风格，展示四层 + 立体判断 + 简历批改。
- 做完起 server 浏览器端到端实测（注意：Windows dev server 历史有 Path/进程坑，用 bounded 方式）。

### D. 后续（核心闭环之后）
生成新简历节点、PDF/Word 导出（Word 的 `lib` + `/api/export-word` MVP 已有，可复用）、咨询对话层（LangGraph human-in-the-loop）、reportId 持久化。
