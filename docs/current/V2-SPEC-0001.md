## 0. 为什么重写

旧 ROADMAP 把 2.0 重心放在"评分稳定系统"——这是**技术债驱动**，从代码缺陷出发。但需求对齐发现：用户真正的、不可替代的价值是 **JD 深度解读**，评分只是支撑性的卫生指标。本文件把基线从"修技术债"校正回"用户价值"。

## 1. 一句话定义

**ResuMatch 2.0 是一个"十年经验 HR + 求职军师"，帮认真求职的人把一份 JD 彻底扒透——看穿岗位真身、判断值不值得投、能不能进、怎么拿下。**

## 2. 目标用户

- 认真找工作、相信"读懂 JD 才是拿到入场券"的求职者。
- 当前真实用户 N=1（owner 本人）。坚持**真实痛点驱动**，不臆造用户。每个功能必须能回指到一次真实求职场景。

## 3. 核心价值（灵魂）

**JD 深度解读**：超越字面，带推理链。

不可替代点（也是对"贴 ChatGPT / 一键改简历竞品"的差异化）：能从 JD 的**命名、措辞、职责软硬**等信号，推到 **公司真实意图 → 岗位真实定位 → HR 潜台词 → 你进去会被放在什么位置**。

标准产出结构（demo 验证过的四层骨架）：
1. **真身**——穿透包装，这岗到底是什么。〔客观读 JD，不依赖用户〕
2. **它要什么人**——JD 背后真正的画像。〔客观读 JD，不依赖用户〕
3. **对你意味着什么**——匹配度 + 诚实的方向/风险提醒（有立场，不无脑说"快投"）。〔**依赖"用户意图"**：方向、处境、对此岗的真实态度〕
4. **怎么打**——简历扬长避短 + 面试戳点 + 谈薪话术。〔**依赖"用户意图"**〕

> 关键：①② 只需读懂 JD；③④ 还必须读懂用户——简历是"你有什么"、不是"你想要什么"。这是从"通用工具"变成"懂你的军师"的门槛，也是甩开"一键改简历"竞品的差异化。用户意图采集方案见 §8。

## 4. 核心闭环（2.0 必做）

| # | 模块 | 回答什么 | 实现依赖 |
|---|---|---|---|
| 1 | **JD 深度解读** | 这岗真身 / 要什么人 / 你的定位 | LLM 推理（JD 文本）★灵魂 |
| 2 | **匹配度评估** | 你能不能进、差距在哪 | LLM + 简历 |
| 3 | **拿下打法** | 简历扬长避短 + 面试 + 谈薪话术 | LLM 推理 |
| 4 | **嗅雷** | 有没有坑、是不是苦力岗 | LLM 嗅 JD 文本信号 + 🟢 硬事实企业背景（工商/司法风险，加 tool，结构化可追溯） |

四块共同特征：靠 LLM 推理 + 用户简历/JD + 结构化工商数据即可完成，**不依赖不可控外部数据、不违背"不编造"铁律**。

## 5. 边界（明确不做 / 推后到 backlog）

- 🟡 **软口碑公司分析**（加班、裁员、是否血汗、PUA 风评）：无开放 API、需爬取、易幻觉、撞"可追溯"铁律。推后。
- 🟡 **真实薪资锚点数字**（这岗市场价多少）：需薪资数据库。2.0 只给"谈薪策略/话术"，不给数字。推后。
- ❌ **账号系统 / 登录 / 云端账户体系**：2.0 走轻量路线（reportId + 链接回看即可）。
- ❌ **复杂多 Agent 玩具**：Agent 可分工，但克制，不为炫技拆 agent。

## 6. 铁律（贯穿所有实现）

1. **不编造数据。** 每条结论可追溯到：JD 原文 / 简历原文 / 有来源的结构化工商数据。
2. **评分是卫生指标，不是卖点。** 必须修漂移（实测同输入 58/63/75 不可接受），但不喧宾夺主。实现方向：模型给等级、后端按固定权重算分（承接旧 ROADMAP 阶段 B 合理部分）。
3. **输出结构稳定。** 四层骨架是契约，前端可稳定渲染；不允许 agent 直接控制最终分数或破坏结构。
4. **每条建议必须可定位回简历原文或 JD 要求。**

## 7. 成功长什么样（初版指标，待细化）

- **解读深度**：对一份 JD，能稳定产出四层骨架，人工评判达到"demo 同等深度"（建议建 5–10 份 JD 的解读质量评判集做回归）。
- **评分稳定**：同简历 + 同 JD 多次运行，最终分漂移 ≤ 5 分（阈值待定）。
- **可追溯率**：100% 建议 / 结论可指回原文或有来源数据。
- **闭环完整**：一份 JD + 一份简历 → 四块输出齐全、结构稳定、可导出。

## 8. 技术架构（已定，2026-06-24 读码后拍板）

- **形态**：把现有"单次调用 8 合 1"拆成 **LangGraph(JS 版)编排的多节点 workflow**，跑在现有 Next.js / Vercel 单体里——不引 Python、不拆分布式（Python 版生态优势此复杂度用不上，代价却全付）。
- **重构方式 = 换骨架、移植器官**：保留现有防幻觉 / 逐字定位 / 后端定分 / 结构校验 / 重试等防御资产（Codex 踩坑攒下的），重写的只是"怎么编排"。
- **节点（5 个，单向为主 + 关键处嵌 agent 判断）**：① JD 解读（★灵魂）→ ② 简历证据提取 → ③ 匹配判断（评分在此降级为支撑信号）→ ④ 应对策略生成 → ⑤ 组装输出。节点图见对话或后续 `docs/current/` 架构图。
- **第三个输入维度 = 用户意图**：简历是"你有什么"、JD 是"岗位要什么"，但 §3 后两层还需要"你想要什么"。采集方式 = **混合**：首次建极简画像（2–3 问：目标方向、绝不接受什么）+ 分析具体 JD 时对话追问一句。主要喂 ③④，①② 不依赖它。
- **对话能力**：用 LangGraph 的 human-in-the-loop（中断 → 问用户 → 继续），既做 JD 信息追问，也做用户意图采集。
- **仍悬置**：评分稳定的具体实现（rubric/权重/缓存）留到 SCORE 卡；`QA-0001`（真实 .docx 端到端）无完成证据，带入 2.0。

## 10. 对话层规划（分析后咨询 + 分析前意图采集）· 2026-06-24

产品从"一次性报告"升级为"能持续陪你的军师"的关键。两个对话场景，是同一套 LangGraph human-in-the-loop 能力的两端：

- **分析前**：对话采集用户意图（你想要什么）。
- **分析后**：用户对结果追问（"待多久能转？""SQL 不展开会被刷吗？""帮我改写这段"）。

**核心优势**：咨询 agent 攥着本次分析的完整产出（①②③④ + 用户意图），追问是基于"已做完的深度分析"回答——比用户单独去问 ChatGPT 强得多。

**真 agent 在这里**：咨询 agent 自主判断每个追问是「直接答」还是「调节点重算」（"帮我改写"→调改写动作；"换个 JD"→重跑 pipeline）。

**依赖**：核心 pipeline 完整 + 结果存得住（reportId / 持久化）+ 前端聊天 UI。位置在接前端之后。
**现在埋的伏笔**：每个节点的结构化输出就是咨询的上下文底料（已在产出）；结果要存得住，咨询才能延续。
**要克制**：锚定"本次分析相关" + 限轮数，别变无限免费陪聊（成本）。
