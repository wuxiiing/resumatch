# ResuMatch Project Workflow

## 角色分工

项目负责人 / 监理：麦桐。

- 确认需求边界。
- 提供 API Key、测试简历、测试 JD。
- 判断产品体验是否符合目标。
- 决定是否进入下一阶段。

Codex 总控窗口：贴身秘书和项目总负责人。

- 不做业务代码开发。
- 负责项目规划、任务拆分、风险提醒、验收标准和进度记录。
- 给其他执行窗口生成明确任务卡。
- 根据执行窗口的回传结果做验收判断。
- 有不确定点时先问项目负责人。

执行窗口：施工窗口。

- 一个窗口只执行一个明确任务。
- 只按任务卡工作，不自行扩展范围。
- 完成后回传改动文件、验证结果、问题和下一步建议。

## 命名施工窗口

后续任务卡必须写明交给哪个施工窗口。窗口可以按项目复杂度逐步增加，不需要一次性全部开启。

### 基础窗口

总控室：当前核心窗口。

- 负责规划、任务拆分、风险提醒、验收和下一步决策。
- 不直接做业务代码开发。
- 不替施工窗口私自扩大需求。

侦察窗口：只读检查窗口。

- 负责阅读文档、检查项目结构、检查依赖状态、发现风险。
- 默认不修改文件。
- 适合开工前、返工前、报错不明时使用。

前端施工窗口：界面和交互窗口。

- 负责上传区、JD 输入、报告页、颜色标注展示、分数面板、提示文案。
- 默认不修改 DeepSeek API、文件解析、限流、Word 导出等后端逻辑。

后端施工窗口：服务端和分析链路窗口。

- 负责 API Route、文件解析流程、DeepSeek 调用、JSON 校验、字数限制、IP 限流。
- 默认不修改页面样式和复杂 UI 组件。

导出施工窗口：Word 导出窗口。

- 负责 Word 报告生成、干净文本导出、建议附录。
- 默认不修改主分析接口和主页面结构。

测试验收窗口：检查和复现窗口。

- 负责运行测试、构建检查、复现问题、检查改动范围和验收标准。
- 默认只读或做很小的明确修复，不能自行重构。

### 多窗口协作规则

1. 每个施工窗口一次只接收一张任务卡。
2. 每张任务卡必须写明施工窗口名称、任务目标、允许修改范围、禁止事项和验收标准。
3. 不让两个施工窗口同时修改同一批文件。
4. 涉及共享文件时，必须由总控室指定唯一负责人。
5. 施工窗口完成后，必须先回传总控室验收，再进入下一张任务卡。
6. 如施工窗口发现任务不清楚，必须先提问，不得自行补需求。

## 工作规则

1. 本核心窗口是项目总控室，不直接写业务代码。
2. 需要开发时，由核心窗口生成任务卡，复制到新的执行窗口。
3. 执行窗口完成后，把结果复制回核心窗口验收。
4. 未经确认，不加入登录、付费、多语言、PDF 导出、一键修改等功能。
5. 所有 API Key 只放环境变量，不写进前端代码，不提交到 Git。
6. 每个阶段先跑通主链路，再补边界和体验。
7. 每次完成一个稳定阶段后再考虑 Git 提交。
8. 任务卡必须标明施工窗口名称，不能把多个窗口的工作混在一张卡里。
