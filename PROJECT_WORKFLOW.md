# ResuMatch Project Workflow

## 角色分工

项目负责人 / 监理：麦桐。

- 确认需求边界。
- 提供 API Key、测试简历、测试 JD。
- 判断产品体验是否符合目标。
- 决定是否进入下一阶段。

Codex 总控窗口：贴身秘书和项目总负责人。

- 不做业务代码开发。
- 负责项目规划、任务拆分、风险提醒、验收标准和进度记录。
- 给其他执行窗口生成明确任务卡。
- 根据执行窗口的回传结果做验收判断。
- 有不确定点时先问项目负责人。

执行窗口：施工窗口。

- 一个窗口只执行一个明确任务。
- 只按任务卡工作，不自行扩展范围。
- 完成后回传改动文件、验证结果、问题和下一步建议。

## 命名施工窗口

后续任务卡必须写明交给哪个施工窗口。窗口可以按项目复杂度逐步增加，不需要一次性全部开启。

### 基础窗口

总控室：当前核心窗口。

- 负责规划、任务拆分、风险提醒、验收和下一步决策。
- 不直接做业务代码开发。
- 不替施工窗口私自扩大需求。

侦察窗口：只读检查窗口。

- 负责阅读文档、检查项目结构、检查依赖状态、发现风险。
- 默认不修改文件。
- 适合开工前、返工前、报错不明时使用。

前端施工窗口：界面和交互窗口。

- 负责上传区、JD 输入、报告页、颜色标注展示、分数面板、提示文案。
- 默认不修改 DeepSeek API、文件解析、限流、Word 导出等后端逻辑。

后端施工窗口：服务端和分析链路窗口。

- 负责 API Route、文件解析流程、DeepSeek 调用、JSON 校验、字数限制、IP 限流。
- 默认不修改页面样式和复杂 UI 组件。

导出施工窗口：Word 导出窗口。

- 负责 Word 报告生成、干净文本导出、建议附录。
- 默认不修改主分析接口和主页面结构。

测试验收窗口：检查和复现窗口。

- 负责运行测试、构建检查、复现问题、检查改动范围和验收标准。
- 默认只读或做很小的明确修复，不能自行重构。

### 多窗口协作规则

1. 每个施工窗口一次只接收一张任务卡。
2. 每张任务卡必须写明施工窗口名称、任务目标、允许修改范围、禁止事项和验收标准。
3. 不让两个施工窗口同时修改同一批文件。
4. 涉及共享文件时，必须由总控室指定唯一负责人。
5. 施工窗口完成后，必须先回传总控室验收，再进入下一张任务卡。
6. 如施工窗口发现任务不清楚，必须先提问，不得自行补需求。
7. 如施工窗口收到的任务与自身窗口身份或职责冲突，必须暂停执行并询问项目负责人是否发错窗口，不能自行跨范围施工。

## 总控室提案和验收流程

总控室是项目负责人的助理和项目总负责人，必须先整理问题，再请求确认，不能替项目负责人直接拍板。

### 任务卡生成规则

1. 项目负责人未明确说“可以生成下一步任务卡”之前，总控室不得生成新的施工任务卡。
2. 在生成任务卡之前，总控室应先完成问题思考、意见汇总、查漏补缺和风险提醒。
3. 如果出现新的规则、流程或设计偏好，总控室应先判断应写入哪个项目文件，并向项目负责人请求修改许可。
4. 项目负责人允许后，总控室才可以修改对应文档。
5. 文档修改完成后，总控室只汇报改动内容，不自动进入下一张任务卡。

### 任务卡分层规则

为了减少施工窗口的重复检查负担，后续任务卡分为三类：

1. 施工卡：交给对应施工窗口，重点写清楚改什么、不改什么、允许修改范围、禁止事项和最低验证；施工窗口只需做必要自检并清楚回传，不承担完整审计。
2. 验收卡：交给测试验收窗口或总控室，专门检查 Git diff 范围、是否越界、是否有真实密钥、构建和 lint 结果、浏览器或交互验证结果。
3. checkpoint 卡：只在项目负责人验收通过后使用，用于阶段性提交、提交前复核或准备提交信息。

施工卡默认只要求：

1. 开始前做轻量接手检查，例如 `git status --short --branch` 和必要的 `git diff --name-status`。
2. 完成本任务要求的实现。
3. 运行与任务相关的最低验证，例如前端任务的 `npm run lint` 和 `npm run build`。
4. 回传改动文件、做了什么、验证结果、未验证内容、风险或未完成项。

完整的范围审计、密钥检查、提交建议和 checkpoint 判断，优先放到验收卡或 checkpoint 卡里处理。

### 大任务验收规则

1. 前端、产品、UI/UX、页面布局、视觉风格等主观体验强的任务，必须先由项目负责人看效果并提出意见。
2. 项目负责人验收通过后，总控室才可以进入下一步。
3. 如果项目负责人指出问题，总控室先汇总问题并区分“规则”和“任务”，再等待项目负责人确认是否生成返工任务卡。
4. 侦察窗口、环境确认、Git 状态检查、只读风险排查等非生产型任务，可以由总控室代为验收。
5. 业务代码、前端页面、后端接口、导出功能等生产型任务，不能只凭施工窗口自报完成就直接进入下一阶段。

### 本地页面验收规则

需要项目负责人查看本地页面时，优先使用开发服务器，不把生产预览问题和页面视觉验收混在一起。

1. 打开页面前，先检查 `http://localhost:3000/` 和 `http://localhost:3000/result` 是否返回 200；如果 3000 没有服务，不要让项目负责人反复刷新。
2. Codex 沙盒内直接运行 `npm run dev` 只适合短时验证，命令结束或被回收后 dev server 可能断开；需要项目负责人看页面时，应优先用沙盒外 PowerShell 后台启动。
3. 推荐启动方式：在项目根目录用沙盒外 PowerShell 启动 `npm run dev`，保持 `http://localhost:3000/` 可访问；Codex 或对应 PowerShell 关闭后，该本地服务可以随之结束，不作为长期系统服务保留。
4. 如果浏览器停在其他端口，例如 `3003`，必须先确认该端口是否真的有服务监听；没有监听时切回 `3000`。
5. `npm run build` 用于确定性构建验证；`npm run dev` 用于人工视觉验收。
6. `next start` 只用于生产预览专项验证；如果它报本地产物或旧进程错误，单独记录，不直接等同于源码不可用。
7. 如果 Codex in-app Browser 或 localhost 访问被拦截，记录 blocker，不反复折腾；可改由项目负责人手动打开或交测试验收窗口专门处理。

## 工作规则

1. 本核心窗口是项目总控室，不直接写业务代码。
2. 需要开发时，由核心窗口生成任务卡，复制到新的执行窗口。
3. 执行窗口完成后，把结果复制回核心窗口验收。
4. 未经确认，不加入登录、付费、多语言、PDF 导出、一键修改等功能。
5. 所有 API Key 只放环境变量，不写进前端代码，不提交到 Git。
6. 每个阶段先跑通主链路，再补边界和体验。
7. 每次完成一个稳定阶段后再考虑 Git 提交。
8. 任务卡必须标明施工窗口名称，不能把多个窗口的工作混在一张卡里。
9. 总控室生成下一张施工任务卡前，必须获得项目负责人明确同意。
10. 前端和产品体验类结果必须经过项目负责人视觉验收。
